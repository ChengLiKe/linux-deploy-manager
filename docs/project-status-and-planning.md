## 五、风险评估与应对措施

| # | 风险 | 概率 | 影响 | 应对措施 |
|---|------|------|------|---------|
| R1 | **Phase 2 实现复杂度过高** — 不同 Linux 发行版(apt/yum/apk)的包管理器差异很大，脚本引擎需要处理大量分支 | 高 | 大 | 1) 初期只支持 Debian/Ubuntu + CentOS 8+，其他发行版 fallback 为手动提示；2) 脚本引擎用 map[os] → commands 的结构，而非 if-else 地狱 |
| R2 | **Phase 2 运行时环境安装失败** — nvm/conda/pyenv 在非交互式 SSH session 中行为不同 | 中 | 大 | 1) 使用 `bash -lc` 加载 profile；2) 提供 init script 模板让用户自定义；3) 安装失败时不清除已有工具，只报告错误 |
| R3 | **Phase 3 回滚时的磁盘空间** — 快照文件可能占用大量磁盘 | 中 | 中 | 1) 默认保留最近 5 个快照；2) 每次创建快照前检查磁盘余量(<1GB 时跳过快照并告警)；3) 提供手动清理 API |
| R4 | **Phase 3 健康检查误判** — HTTP 检查可能因启动延迟而误报失败 | 高 | 中 | 1) 支持配置启动等待时间（默认 5s）；2) 重试机制（默认 3 次）；3) 支持 TCP 和 Process 多种检查模式 |
| R5 | **Phase 4 Webhook 安全** — Webhook URL 和 Secret 在传输中可能泄露 | 中 | 大 | 1) Secret 加密存储（用 AES-GCM）；2) 生成 HMAC-SHA256 签名；3) 不支持 HTTPS 的 URL 给出警告 |
| R6 | **一体机开发资源紧张** — 当前所有工作在一台 Windows 机器上完成，编译/测试串行 | 低 | 中 | 1) 阶段性提交到 GitHub 利用 CI 做并行验证；2) 文档和代码实现交替进行 |
